# Crave 🍔

A cross-platform food delivery app built with **Flutter** and **Firebase** — Google Sign-In, dynamic menu &amp; cart, real-time order tracking, in-app + local notifications, and Stripe payments.

---

## Tech stack

| Area | Choice |
|---|---|
| Framework | Flutter (Material 3, light/dark theme follows system) |
| Architecture | Clean Architecture (feature-first: `data` / `domain` / `presentation`) |
| State management & navigation | [GetX](https://pub.dev/packages/get) |
| Backend | Firebase (Auth, Cloud Firestore, Storage) |
| Payments | [flutter_stripe](https://pub.dev/packages/flutter_stripe) |
| Local notifications | [flutter_local_notifications](https://pub.dev/packages/flutter_local_notifications) |
| Networking | Dio |
| Local storage | SharedPreferences, flutter_secure_storage |
| Responsive UI | flutter_screenutil |
| Env config | flutter_dotenv |

## Project structure

```text
lib/
├── main.dart                  # entry point: native splash, AppConfig.init(), runApp
├── firebase_options.dart      # generated by FlutterFire CLI
└── src/
    ├── app.dart                # GetMaterialApp, theming, routes, initial binding
    ├── config/
    │   └── app_config.dart     # Firebase/Stripe/dotenv/local-notifications init
    ├── features/
    │   ├── auth/                # Google Sign-In, session state
    │   ├── products/            # menu, categories, product detail
    │   ├── cart/                 # cart items, quantities, totals
    │   ├── orders/                # place order, live order tracking, history
    │   ├── payment/               # Stripe payment sheet flow
    │   ├── notifications/         # in-app notification center + unread badge
    │   ├── dashboard/              # bottom-nav shell (Home/Cart/Orders/Profile)
    │   └── profile/                # user profile screen
    ├── routing/
    │   ├── app_routes.dart        # route name constants
    │   ├── app_router.dart        # GetPage list (route -> screen + binding)
    │   └── app_bindings.dart       # globally-registered controllers (fenix: true)
    ├── services/                   # singletons: LocalNotificationService, etc.
    ├── shared/                      # AppStrings, shared widgets, helpers
    ├── theme/                        # AppColors, AppTextStyles, AppSpacing, etc.
    └── utils/                          # runTask/Either helpers, AppLogger, toast
```

Each feature under `lib/src/features/<feature>/` follows the same three layers:

- **`domain/`** — pure Dart: entities, repository *contracts* (abstract classes), use cases (single-responsibility, callable classes).
- **`data/`** — Firestore datasources, models (`fromMap`/`toMap`), repository *implementations* of the domain contracts.
- **`presentation/`** — GetX controllers (`*_controller.dart`), screens, widgets, and bindings.

**Dependency rule:** `presentation` → `domain` ← `data`. Presentation never talks to Firestore directly — always through a use case → repository → datasource.

## How everything fits together (request lifecycle)

1. **App boot** (`main.dart`)
   - `FlutterNativeSplash.preserve()` keeps the native splash on screen.
   - `AppConfig.init()` initializes Firebase, Google Sign-In, loads `.env`, configures Stripe (if a publishable key is present), and initializes `LocalNotificationService`.
   - `App` widget builds `GetMaterialApp` with `AppBindings()` as the **global** initial binding — this registers controllers that must exist app-wide (session, cart, orders, notifications) with `fenix: true` (recreated lazily if disposed).

2. **Routing** (`lib/src/routing/`)
   - `AppRoutes` — string constants for every route.
   - `AppRouter.getPages` — maps each route to a screen + (optional) per-route `Binding` for screen-local controllers.
   - Screens navigate via `Get.toNamed(...)`, `Get.until(...)`, etc. — no second navigation package.

3. **Auth** (`features/auth`)
   - `AuthService` (in `services/`) wraps Firebase Auth + Google Sign-In.
   - On successful Google sign-in, `AuthService` writes/merges a `users/{uid}` profile document (`displayName`, `email`, `photoUrl`, `updatedAt`) — this is what makes the `users` collection exist for a brand-new account.
   - `SessionController` (registered globally) exposes `user` (`Rx<AppUser?>`) and `status` (`SessionStatus`). It bootstraps with `getCurrentUser()` then subscribes to `authStateChanges` for live updates.
   - Other controllers (`OrderController`, `NotificationController`) use `ever(session.user, ...)` to (re)subscribe to per-user Firestore streams whenever the signed-in user changes — this avoids a race condition where a brand-new user's first order/notification stream wouldn't start because `session.user` was still `null` at `onInit()`.

4. **Menu / Products** (`features/products`)
   - `ProductMenuController` loads categories + products from Firestore (`categories`, `products` collections) and exposes search/category filters.
   - `MenuScreen` shows the menu, a category pill bar, food cards, and a notification bell with a live unread-count badge (`Obx` reading `NotificationController.unreadCount`).

5. **Cart** (`features/cart`)
   - `CartController` (global) holds cart items, quantities, and totals (`totalItems`, `totalPrice`).
   - Adding/removing/updating items goes through use cases → `CartRepositoryImpl` (local persistence).
   - The dashboard's cart tab badge reads `cartController.totalItems` reactively.

6. **Checkout & Payment** (`features/payment` + `features/orders`)
   - `PaymentController.pay()`:
     - Guards against double-taps (re-entrancy guard on `status`).
     - Opens the Stripe Payment Sheet.
     - On success: calls `OrderController.placeOrder()`, which writes a new document to the top-level `orders` collection (`userId`, `items`, `totalAmount`, `status: pending`, `createdAt`) and clears the cart.
     - Fires in-app + local notifications ("Payment Successful", "Order Placed") via `NotificationController.createNotification()`.
     - Navigates back to the existing `Dashboard` route (`Get.until`, not `Get.offAllNamed` — preserves `DashboardController`/`CartController` state) with the **Orders** tab selected, then pushes `OrderDetailScreen` for the new order.

7. **Orders** (`features/orders`)
   - `OrderController` (global) subscribes to a live Firestore query: `orders` where `userId == currentUser.id`, ordered by `createdAt`.
   - `orders` (`RxList<FoodOrder>`) drives the Orders tab and Order Detail screen — any Firestore write (new order, status change) reflects immediately via the stream.
   - When an order's `status` transitions (e.g. `pending` → `preparing` → `out_for_delivery` → `delivered`), `_notifyStatusChanges()` detects the diff and fires an "Order status updated" notification.

8. **Notifications** (`features/notifications`)
   - Firestore structure: `users/{userId}/notifications/{notificationId}` — each doc has `title`, `body`, `type`, `isRead`, `createdAt`.
   - `NotificationController` (global) subscribes to a live stream of the current user's notifications, exposes `notifications` (`RxList`) and `unreadCount`.
   - `createNotification()` writes the Firestore doc **and** shows a system notification via `LocalNotificationService` (foreground/background while the app process is alive).
   - `NotificationsScreen` groups notifications into "New" / "Earlier" sections and supports mark-as-read / mark-all-as-read.
   - Malformed Firestore documents (e.g. manually edited in the console) are skipped individually rather than breaking the whole list.

9. **Dashboard** (`features/dashboard`)
   - `DashboardScreen` uses an `IndexedStack` + `DashboardController.selectedIndex` for the bottom nav: `0` Home/Menu, `1` Cart, `2` Orders, `3` Profile.
   - Switching tabs does not rebuild controllers — state (cart contents, order list, scroll position) is preserved.

## State management conventions (GetX)

- Controllers extend `GetxController`; reactive state uses `.obs` / `Rx` / `RxList` / `Rxn`.
- **Global** controllers (session, cart, orders, notifications, auth) are registered once in `AppBindings` with `Get.lazyPut(..., fenix: true)` — `fenix: true` means GetX recreates the controller if it's ever disposed, so `Get.find<T>()` is always safe.
- **Screen-local** controllers are registered via that route's `Binding` in `app_router.dart`.
- `Obx(() => ...)` rebuilds depend on reactive reads happening **synchronously inside the builder** — not inside lazy callbacks like `ListView.builder`'s `itemBuilder` (read the value into a local variable first if needed).
- Async results use `FutureEither<T>` / `StreamEither<T>` (fpdart `Either<Failure, T>`), produced via `runTask()` in datasources/services. Controllers `.fold()` the result: success path updates `.obs` state, failure path calls `showGlobalToast(message: ..., status: 'error')`.

## Firestore data model

| Collection | Path | Notes |
|---|---|---|
| Users | `users/{userId}` | Created/merged on Google sign-in: `displayName`, `email`, `photoUrl`, `updatedAt` |
| Notifications | `users/{userId}/notifications/{id}` | `title`, `body`, `type`, `isRead`, `createdAt` |
| Categories | `categories/{id}` | Menu categories |
| Products | `products/{id}` | Menu items |
| Orders | `orders/{id}` | `userId`, `items[]`, `totalAmount`, `status`, `createdAt` — queried by `userId` |

The app uses a named Firestore database (`food-delivery-db`), configured in `AppConfig.firestore`.

## Notifications

Two layers, both triggered from the same events (order placed, payment success, order status change):

1. **In-app**: written to `users/{userId}/notifications`, shown on the `NotificationsScreen`, and reflected in the bell-icon unread badge on the menu screen.
2. **Local/system** (`flutter_local_notifications`): shown immediately while the app process is alive (foreground or backgrounded), via `LocalNotificationService`.

## Getting started

### Prerequisites

- Flutter SDK (>= 3.5.0)
- A Firebase project with Auth (Google), Cloud Firestore, and Storage enabled
- A Stripe account (test keys are fine for development)

### Setup

1. Install dependencies:
   ```bash
   flutter pub get
   ```
2. Add Firebase config:
   - `android/app/google-services.json`
   - `ios/Runner/GoogleService-Info.plist`
   - `lib/firebase_options.dart` (generate via `flutterfire configure`)
3. Create a `.env` file in the project root (not committed — see `.gitignore`):
   ```env
   STRIPE_PUBLISHABLE_KEY=pk_test_xxx
   ```
4. Run the app:
   ```bash
   flutter run
   ```

### Release builds (Android)

Release builds are signed using a keystore stored **outside** the repo, referenced via `android/key.properties` (gitignored). See your local setup notes for the keystore location and SHA fingerprints — the SHA-1/SHA-256 of the release keystore must be registered in the Firebase console (Project Settings → Your app → SHA certificate fingerprints) for Google Sign-In to work in release builds.

```bash
flutter build apk --release
```
